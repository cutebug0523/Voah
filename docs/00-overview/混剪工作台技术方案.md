# Codex 混剪工作台技术方案

## 1. 目标

在 Codex 内实现一套“给视频素材，生成混剪成片”的工作流能力。

这不是新起一个独立产品盘，也不是先做完整 SaaS，而是把能力沉淀成：

- **Codex Skill**：定义工作流、提示词、决策边界、产物格式和人工确认点。
- **MCP 工具**：封装可重复调用的能力，如素材索引、向量检索、ASR、TTS、字幕生成、FFmpeg 渲染。
- **本地脚本**：处理确定性任务，如切镜头、抽帧、音频提取、时间线渲染。
- **素材库**：缓存已理解的视频片段，避免重复处理。

最终使用体验应接近：

```text
用户：用这些素材剪一条 60 秒情绪向短视频，风格参考“悬疑、递进、最后反转”

Codex：
1. 扫描并理解素材
2. 检索可用片段
3. 生成文案和剪辑时间线
4. 询问是否确认方案
5. 调用 TTS、字幕、FFmpeg 合成
6. 输出成片和中间工程文件
```

## 2. 核心原则

### 2.0 Voah 工程化底座

本项目按 `Voah 系列工程化底座` 推进：

```text
常驻素材库
  -> 单次任务流水线
```

`voah-video-intake` 是常驻素材库建设，不属于每次混剪任务正文。当前 Voah 主线从任务 brief / 产品卖点 / 销售逻辑开始，先生成连续口播和 TTS/audio_sections，再按口播语义召回素材；每一步都必须有结构化产物承接上一环并导向下一环。

详见：

```text
docs/00-overview/Voah系列工程化底座.md
```

### 2.1 LLM 不直接剪视频

LLM 负责理解目标、组织叙事、选择素材、生成文案和时间线方案。

真正的视频切割、拼接、混音、字幕烧录由 FFmpeg 等确定性工具执行。

### 2.2 原始素材不重复理解

素材一旦入库，应缓存：

- 文件 hash
- 视频元信息
- 镜头/语义片段
- 关键帧
- ASR 转写
- OCR 文本
- 画面摘要
- 向量 embedding
- 模型版本和处理参数

后续再次使用相同素材时，只增量处理变化部分。

### 2.3 时间线是系统核心中间产物

所有剪辑决策最终都要落成结构化时间线。

不要只保存自然语言脚本。自然语言脚本适合人看，时间线 JSON 适合工具执行。

### 2.4 向量召回不是最终剪辑决策

视频素材入库后会做向量化，但向量检索只负责“粗召回候选片段”，不能直接决定最终剪点。

真正可用的剪辑链路应是：

```text
用户需求 / 文案意图
  ↓
向量召回候选 asset / scene / transcript / ocr / frame
  ↓
模型 temporal rerank
  ↓
重新打可用时间点 usable_start / usable_end
  ↓
代码校验时间点并 snap 到镜头、关键帧或句子边界
  ↓
FFmpeg 按时间线切割重构
```

这里的 rerank 不是普通文本 rerank，而是“时间局部化 + 剪辑可用性判断”。

模型需要回答的不只是“这个片段相关吗”，还要回答：

```json
{
  "source_path": "/absolute/path/source.mp4",
  "candidate_start": 4.0,
  "candidate_end": 11.0,
  "score": 0.87,
  "usable_start": 4.9,
  "usable_end": 7.2,
  "role": "证据展示",
  "reason": "这里是口红上嘴特写，能证明丝滑质地。",
  "cut_padding": {
    "before": 0.2,
    "after": 0.3
  },
  "risks": ["底部有原硬字幕"]
}
```

最终时间点必须由代码二次校验，包括：

- 起止时间不能越界。
- 片段时长满足目标段落。
- 尽量贴近镜头边界、关键帧边界或 ASR 句子边界。
- 避免切在动作中间、话没说完、字幕闪烁处。
- 根据硬字幕位置决定 keep / crop / cover / avoid。

### 2.5 人类保留关键审美决策

MVP 阶段不追求完全自动化。关键节点应允许人工确认：

- 主题和风格
- 文案
- 片段选择
- 配音声线
- 字幕样式
- 原素材硬字幕处理策略
- 最终时间线

## 3. 总体架构

```text
视频素材
  ↓
素材扫描与去重
  ↓
镜头切分 / 抽帧 / 音频提取 / ASR / OCR / 画面理解
  ↓
片段结构化库 + 向量库 + 原文件索引
  ↓
用户输入主题、风格、时长、平台、约束
  ↓
检索候选片段
  ↓
LLM 生成文案 + 剪辑时间线
  ↓
人工确认或修改
  ↓
TTS 配音
  ↓
字幕对齐 / 原硬字幕避让或遮挡
  ↓
FFmpeg 渲染
  ↓
成片 + 时间线 + 字幕 + 工程缓存
```

## 4. Skill 与 MCP 的边界

### 4.1 适合做成 Codex Skill 的部分

Skill 负责“怎么协作”和“怎么判断”。

建议先做一个主 Skill：

```text
video-remix
```

触发语义：

- “混剪这个素材”
- “用这些视频剪一条”
- “生成配音字幕”
- “给我出一版短视频”
- “按某种风格重剪”

Skill 内应定义：

- 素材准备流程
- 需求澄清问题
- 风格分析方法
- 文案生成规则
- 时间线 JSON 格式
- 审核清单
- 渲染前确认点
- 常见失败处理方式

Skill 不应承载大量业务代码。代码应放到 scripts 或 MCP 工具里。

### 4.2 适合做成 MCP 的部分

MCP 负责“可被 Codex 调用的能力”。

建议拆成几个工具域：

```text
media_index
  - scan_assets
  - fingerprint_asset
  - get_asset_metadata
  - list_indexed_clips

media_understand
  - detect_scenes
  - extract_keyframes
  - transcribe_audio
  - run_ocr
  - describe_frames
  - embed_clip

media_retrieve
  - search_clips_by_text
  - search_clips_by_image
  - filter_clips_by_duration
  - filter_clips_by_tags

voice_subtitle
  - synthesize_voice
  - align_subtitles
  - generate_srt
  - generate_ass
  - detect_burned_subtitle_region
  - choose_subtitle_policy

media_render
  - validate_timeline
  - render_preview
  - render_final
  - export_project_bundle
```

### 4.3 适合做成本地脚本的部分

脚本用于稳定、低变动、可测试的处理逻辑：

- FFmpeg/ffprobe 包装
- scene detection
- keyframe extraction
- subtitle file generation
- timeline validation
- render command generation
- cache cleanup

这些脚本可以先由 Skill 直接调用；当接口稳定后，再包成 MCP。

## 6. 推荐实现阶段

### 阶段 1：文档和协议

目标：先统一工作流和数据结构。

产物：

- 本方案文档
- `video-remix` Skill 草案
- timeline JSON schema
- clip metadata JSON schema
- 渲染前检查清单

不做：

- 不实现完整 MCP
- 不做 UI
- 不接复杂云服务

### 阶段 2：本地最小闭环

目标：只用本地文件和脚本跑通一条视频。

能力：

- 扫描素材目录
- ffprobe 读取元信息
- 优先验证阿里百炼 Qwen-Omni HTTP 视频理解结果能否转成内部 Clip Metadata
- FFmpeg 抽音频
- FFmpeg 按固定间隔抽帧
- 简单镜头切分
- 手动或半自动写片段摘要
- LLM 生成时间线 JSON
- FFmpeg 根据时间线渲染成片

暂时可不做：

- 向量库
- 自建全量抽帧图片理解
- 自动 TTS
- 自动字幕动效

### 阶段 3：素材理解和缓存

目标：让素材库真正可复用。

能力：

- 文件 hash 去重
- ASR 转写
- OCR
- 关键帧画面描述
- clip metadata 入库
- 简单向量检索

建议优先实现结构化库，再加向量库。

结构化库可以先用 SQLite；向量能力可以先用本地文件或 SQLite 扩展，接口稳定后再换专用向量库。

### 阶段 4：TTS、字幕和节奏

目标：让成片具备可发布雏形。

能力：

- TTS 生成配音
- 根据 TTS 时间戳生成 SRT/ASS
- 字幕样式模板
- 检测原素材是否有硬字幕
- 支持 `keep` / `crop` / `cover` 三种硬字幕处理策略
- BGM 混音
- 口播时长和画面时长自动校准

### 阶段 5：MCP 化

目标：把稳定脚本变成可复用工具。

优先 MCP 化：

1. `media_index.scan_assets`
2. `media_render.validate_timeline`
3. `media_render.render_final`
4. `voice_subtitle.generate_ass`
5. `media_retrieve.search_clips_by_text`

不建议一开始就全部 MCP 化。先让脚本跑稳，再封装。

## 7. Codex Skill 草案

### 7.1 Skill 名称

```yaml
name: video-remix
description: 用于在 Codex 内完成视频素材理解、混剪文案生成、时间线规划、配音字幕和 FFmpeg 渲染协作。当用户要求“混剪视频”“用素材生成短视频”“加配音字幕”“按某种风格重剪”时触发。
```

### 7.2 Skill 工作流

```text
1. 明确需求
   - 成片平台
   - 目标时长
   - 横竖屏比例
   - 风格
   - 是否需要配音
   - 是否需要字幕
   - 是否有参考视频/参考文案

2. 检查素材
   - 找到素材目录
   - 列出视频文件
   - 检查是否已入库
   - 对未入库素材建立索引

3. 生成片段清单
   - 切分镜头
   - 提取关键帧
   - 整理 ASR/OCR/画面摘要

4. 生成创意方案
   - 输出 1-3 个方向
   - 每个方向包含标题、钩子、结构、预计时长
   - 等用户选择

5. 生成文案和时间线
   - 先出自然语言脚本
   - 再出 timeline JSON
   - 验证时间线可执行

6. 生成音频字幕
   - TTS
   - 字幕对齐
   - 生成 SRT/ASS

7. 渲染
   - 预览版
   - 用户确认
   - 最终版

8. 归档
   - 成片
   - 时间线 JSON
   - 字幕文件
   - 配音文件
   - 处理日志
```

### 7.3 Skill 的关键约束

- 不要直接覆盖原始素材。
- 渲染前必须验证 timeline。
- 对长耗时处理先说明将执行什么。
- 不主动启动服务。
- 不在未确认时删除缓存。
- 遇到素材版权、真人声音克隆、敏感内容时，先提示风险和边界。

## 9. FFmpeg 渲染边界

FFmpeg 适合负责：

- 裁剪片段
- 拼接片段
- 转码
- 横竖屏转换
- 音频混音
- 字幕烧录
- 原硬字幕区域裁剪、遮挡或局部模糊
- 加背景音乐
- 基础滤镜

FFmpeg 不适合负责：

- 理解素材含义
- 准确识别硬字幕语义和位置
- 判断故事结构
- 自动写文案
- 审美取舍
- 判断平台传播策略

## 10. MVP 任务拆分

建议接下来按这个顺序一个一个实现：

1. 创建 `video-remix` Skill 骨架。
2. 定义 `clip.schema.json` 和 `timeline.schema.json`。
3. 写 `scripts/probe_media.py`：读取视频元信息。
4. 写 `scripts/extract_audio.py`：提取音频。
5. 写 `scripts/extract_keyframes.py`：抽关键帧。
6. 写 `scripts/validate_timeline.py`：校验时间线。
7. 写 `scripts/render_timeline.py`：用 FFmpeg 渲染时间线。
8. 再考虑 ASR、TTS、向量库和 MCP 化。

这个顺序的好处是：先把“时间线能不能执行”跑通，再逐步增强“时间线怎么生成”。

## 11. 当前决策

- 先不新建独立应用。
- 先不做前端 UI。
- 先不做完整 MCP。
- 先在 Codex 工作区内沉淀 Skill、schema、脚本和文档。
- 先从本地 FFmpeg 闭环开始。
- 视频理解优先验证阿里百炼 Qwen-Omni HTTP 非实时调用；抽帧 + ASR 作为 fallback。

## 12. 待确认问题

后续实现前需要逐步确认：

1. 素材库默认放在当前工作区，还是放在用户统一媒体库目录？
2. 第一个 MVP 是否只支持本地视频文件？
3. 阿里百炼视频理解是否要求素材先上传 OSS，还是支持直接 URL/本地上传？
4. 成片默认比例是 `9:16` 还是跟随原素材？
5. 配音第一版使用哪类 TTS：系统语音、本地模型，还是第三方 API？
6. 原素材硬字幕默认策略使用 `keep`、`crop` 还是 `cover`？
7. 是否需要保留“人工精修时间线”的文本格式，方便你手改？
