# Voah 桌面生产端前端设计方案

> 状态：设计方案，待评审。本文只定义前端，不改动 CLI 生产内核。
> 前置：CLI 生产内核已落地（见 [Voah-CLI化生产内核方案](./Voah-CLI化生产内核方案.md)），所有生产逻辑（状态机、OSS、stale、QA gate、worker 编排）都在 `voah` CLI 里，前端不重写。

## 1. 目标与定位

一句话：**给员工用的批量出片工作台，面向一天 150 条。**

设计三原则：

- **低心智优先**：日常走量路径越短越好。选产品 + 数量 + 时长就能开跑。
- **可控不失控**：要质量时能预览文案、试听配音、单条复跑，但这些是"展开"出来的，不挡在走量路径上。
- **前端是壳**：前端只做提交参数、调 `voah` 命令、读 manifest/日志、展示状态。不碰生产逻辑。

非目标：

- 不做权限/多租户（单机生产工具）。
- 不在前端重写任务状态机、OSS 上传、stale 判断、QA gate。
- 不做素材精剪时间线编辑器（那是另一类工具）。

## 2. 两档心智模式

这是化解"低心智"与"带预览微调"张力的核心设计。

```text
快车道（默认，走量）
  选产品 -> 填数量 -> 选时长 -> 开跑
  -> 直接 voah batch run，进队列看板盯进度

精修道（按需展开）
  选产品 -> 生成 1 条样片的文案+配音 -> 预览微调 -> 满意后批量复制参数开跑
  -> 适合新产品首次、或某产品文案没调顺时
```

员工 90% 时间在快车道。精修道是一个"先打样再放量"的入口，不是每条都走。

## 3. 信息架构

四个主区域，左侧固定导航：

```text
┌────────────┬──────────────────────────────────────────┐
│            │                                          │
│  ▸ 队列     │   主工作区（随导航切换）                    │
│  ▸ 产品     │                                          │
│  ▸ 成品库   │                                          │
│  ▸ 设置     │                                          │
│            │                                          │
│  ──────    │                                          │
│  今日 32/150│                                          │
│  运行 3     │                                          │
│  待审 5     │                                          │
└────────────┴──────────────────────────────────────────┘
```

- **队列**（主画面，打开即见）：今日批量任务看板 + 新建批量入口。
- **产品**：产品库管理 + 素材入库 + 精修打样。
- **成品库**：合格成片浏览、人工复核、导出清单。
- **设置**：API key、产品默认参数、TTS 音色、字幕样式。

左下角常驻今日产能简报（出片数/上限、运行中、待审），不用进 dashboard 也能瞥见全局。

## 4. 关键页面

### 4.1 队列看板（主画面）

```text
┌─────────────────────────────────────────────────────────┐
│  今日队列                              [ + 新建批量 ]      │
│  ───────────────────────────────────────────────────     │
│  今日 32/150   运行 3   待审 5   失败 2   完成 22          │
│                                                           │
│  批次 huaxizi-qidian · 45秒抖音 · 20 条      14:30 创建     │
│  ▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░░░  12/20    [暂停] [详情]              │
│   ├ #03 文案 ▸ 配音 ▸ 召回 ▸ 渲染 ▸ QA   ✓ 完成            │
│   ├ #04 文案 ▸ 配音 ▸ 召回 ▸ 渲染 ▸ QA   ⟳ 渲染中          │
│   ├ #05 文案 ▸ 配音 ▸ 召回 ▸ 渲染 ▸ QA   ⚠ 待审            │
│   └ #06 文案 ▸ 配音 ▸ 召回 ▸ 渲染 ▸ QA   ✗ 失败 [重试]     │
│                                                           │
│  批次 fangshai-qidian · 30秒 · 12 条        11:00 创建     │
│  ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  12/12    ✓ 全部完成  [导出]         │
└─────────────────────────────────────────────────────────┘
```

要点：

- 每条任务用 5 段阶段条（文案/配音/召回/渲染/QA）显示当前进度，颜色编码状态。
- 失败任务原地 `[重试]`，对应 `voah task run {dir} --from {failed_stage}`。
- 批次级 `[暂停]`（停止派发新任务，不杀正在跑的）。
- 一眼看出"哪条卡住了、卡在哪一步"，这是 150 条/天盯盘的核心。

### 4.2 新建批量（快车道）

```text
┌──────────────────────────────────┐
│  新建批量                          │
│  ────────────────────────────     │
│  产品    [ 花西子气垫        ▾ ]    │
│  数量    [  20  ] 条               │
│  时长    ( )15秒 (•)45秒 ( )自定义  │
│                                    │
│  ▸ 高级（平台/文案/音色…）  默认收起 │
│                                    │
│         [ 打样 1 条 ]  [ 直接开跑 ] │
└──────────────────────────────────┘
```

- 三个必填：产品、数量、时长。其余全用产品库默认 + 全局设置，收在"高级"里。
- `[直接开跑]` → `voah batch run`，立刻进队列。
- `[打样 1 条]` → 进精修道（见 4.3）。

### 4.3 精修打样（精修道）

```text
┌─────────────────────────────────────────────┐
│  打样 · 花西子气垫 · 45秒                       │
│  ─────────────────────────────────────────   │
│  文案稿                          [重新生成]     │
│  ┌─────────────────────────────────────┐     │
│  │ 01 你还在为底妆卡粉烦恼吗…   [编辑]    │     │
│  │ 02 花西子气垫,上脸服帖…     [编辑]    │     │
│  └─────────────────────────────────────┘     │
│                                               │
│  配音                  ▶ 0:42  [试听] [换音色]  │
│                                               │
│  ─────────────────────────────────────────   │
│  样片满意?  [ 用这套参数批量跑 20 条 ]          │
└─────────────────────────────────────────────┘
```

- 跑 `voah copy run` + `voah tts run` 出 1 条样片的文案和配音。
- 文案可编辑（写回 voice_script.json → 触发下游 stale，已由 CLI 支持）。
- 配音可试听（`voah tts preview`）、换音色。
- 满意后把这套参数固化进 batch brief，`voah batch run`。

### 4.4 产品页

- 产品列表 + 详情编辑（产品名、品牌、卖点库、禁忌、CTA、活动）。
- 素材入库：选源目录 + 数量 → `voah intake run`，显示入库记录和状态。
- 产品状态：待入库 / 就绪 / 入库失败。

### 4.5 成品库

- 合格成片网格：缩略图 + QA 摘要 + 最终时长。
- 内嵌播放器看成片。
- 人工复核：通过 / 复查 / 退回（写回本地审核记录）。
- 批次导出合格清单。

### 4.6 设置

- API key 管理（7 个模型模块，masked 显示，对接 `voah config`）。
- 生产默认参数：文案风格/受众/禁忌/CTA、TTS 音色/语速/情绪、字幕样式。
- 这些作为 batch 默认值快照。

## 5. 与 CLI 的映射

前端每个操作都落到一条 `voah` 命令：

```text
新建批量(直接开跑)   -> voah batch run --product .. --count .. --target-duration ..
打样 1 条            -> voah task create + voah copy run + voah tts run
重新生成文案         -> voah copy run {task_dir}
编辑文案后           -> 写 voice_script.json,voah task run {dir} --from tts(CLI 自动 stale)
试听配音            -> voah tts preview --text ..
换音色后批量         -> voah batch run（参数带 voice-id）
失败重试            -> voah task run {dir} --from {failed_stage}
素材入库            -> voah intake run --product .. --source-dir ..
读取队列/进度        -> 读 batch_manifest.json + 各 task_manifest.json
读取日志            -> tail task_dir/logs/{stage}.jsonl
保存 key            -> voah config set ..
看成片/QA           -> 读 full_pipeline_manifest.json + final_subtitled.mp4
```

前端不实现这些命令背后的任何逻辑，只拼参数、起进程、读产物。

## 6. 数据流

```text
前端 UI
  │ 用户操作
  ▼
调用层（spawn voah CLI 或 HTTP）
  │ voah xxx run ...
  ▼
voah CLI（真源头）
  │ 写产物 + manifest + logs
  ▼
本地文件系统
  cache/voah_batches/.../batch_manifest.json
  cache/voah_tasks/.../task_manifest.json + logs/*.jsonl
  │ 前端轮询/监听
  ▼
前端读取 manifest + 日志 -> 渲染队列/进度/产物
```

关键：**前端状态以 manifest 文件为真源**，不在前端内存里维护第二套任务状态。CLI 写什么，前端读什么。这样 CLI 命令行单独跑和前端跑，状态完全一致。

进度刷新策略：

- 运行中批次：轮询 `task_manifest.json`（每 1-2s）或用文件监听（fs.watch）。
- 日志：跟读 `logs/{stage}.jsonl` 增量行。

## 7. 技术栈

CLI 是零依赖 Node，产物是本地文件。前端需要：调本地 CLI、读本地文件、播本地视频。

### 方案对比

| 维度 | A. Electron + 主进程调 CLI | B. voah 起 HTTP server + 浏览器 |
|---|---|---|
| 调 CLI | 主进程 spawn `voah`，IPC 给渲染层 | CLI 包一层 HTTP，前端调 REST |
| 读本地产物 | 直接读文件系统 | 需后端代理文件接口 |
| 播本地视频 | `file://` 直接播 | 需后端做静态服务/range 请求 |
| 多人共享一台机 | 不便（单机单窗） | 方便（多浏览器连一台） |
| 与现状一致性 | 现有就是 Electron，迁移平滑 | 要新写 HTTP 层 |
| 打包分发 | 一个 app | 要管服务进程 |
| 复杂度 | 低 | 中（多一层 server） |

### 推荐：A. Electron + 主进程调 CLI

理由：

- 你的场景是**本地批量生产工具**，不是多人共享 SaaS。Electron 直接读本地 cache、`file://` 播成片，最顺。
- 现有桌面端就是 Electron，主进程已有 spawn 框架，把"spawn python 编排"换成"spawn voah CLI"即可，迁移成本低。
- 渲染层全新写：React + Vite（保留现有技术选型，丢弃旧 App.jsx）+ Tailwind（你偏好）+ 组件化拆分 + 轻量状态（zustand 或 Context）。

如果将来要多人共享一台生产机，再演进到 B（CLI 加 HTTP 层），前端展示层可大部分复用。

### 渲染层结构建议

```text
desktop/voah-studio/src/
  app/            路由 + 布局（左侧导航 + 主工作区）
  pages/
    QueuePage      队列看板（主画面）
    ProductPage    产品 + 入库 + 打样
    OutputPage     成品库 + 复核
    SettingsPage   设置
  features/
    batch/         新建批量、打样
    task/          任务进度、阶段条、重试
    review/        成品复核
  components/      Field/Badge/Metric/StageBar/VideoPlayer/LogViewer（可复用）
  hooks/           useManifest（读+轮询）、useBatch、useSettings
  ipc/             voahClient（封装所有 voah 命令调用）
```

主进程只保留：spawn `voah`、文件读取、fs.watch、`file://` reveal。生产编排全部由 CLI 承接。

## 8. 从现有桌面端迁移

保留：

- Field/Badge/Metric/EmptyState 等原子组件（重新整理）。
- 当前 `desktop/voah-studio` 的 CLI-first IPC、任务中心、设置页和产品库能力。

重写：

- 只在 `desktop/voah-studio` 内迭代页面与组件。
- 不再维护历史实验壳。

删除：

- 历史实验壳及其本地编排/contract 路线。
- 桌面端不得再保留第二套生产编排，生产真源统一是 `voah` CLI。

## 9. 分期建议

- **M1 走量闭环**：队列看板 + 新建批量(快车道) + 调 `voah batch run` + 进度轮询 + 失败重试。能从空白到批量出片。
- **M2 质量闭环**：精修打样(文案预览/编辑、配音试听/换音色) + 成品库复核 + 导出清单。
- **M3 打磨**：产品库完整编辑、入库界面、设置面板、日志查看器、产能简报。

M1 做完就能投入日常走量，M2/M3 提升可控性和体验。
