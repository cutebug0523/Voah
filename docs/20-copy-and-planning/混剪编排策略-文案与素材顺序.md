# 混剪编排策略：文案先行、素材先行与混合方案

## 1. 当前结论

短视频带货混剪的当前主线已经定为：先定全片销售逻辑，写连续口播，生成 TTS，再按口播语义分段去召回、重打点和填充素材。

也就是：

```text
先定全片销售逻辑
  -> 写连续口播
  -> TTS
  -> 按口播语义分段
  -> 每段召回/填充素材
  -> 字幕用口播原文断句
```

素材约束仍然重要，但它发生在“按口播语义召回/重打点/填充”阶段，而不是先固定一排 shot 再逐 shot 写稿。

素材填充规则：

```text
素材宜长不宜短。
长素材可剪。
短素材优先用同语义/同维度片段拼接。
不要因为只差一点点就默认循环凑够。
```

以下旧的“混合方案”仍保留作为推理背景，但主线以本节为准。

## 1.1 历史讨论：混合方案背景

短视频带货混剪不适合纯“先文案”，也不适合纯“先视频”。

在工程链路上，`voah-video-intake` 属于常驻素材库；一次具体混剪任务从召回产物开始。也就是说，文案阶段不直接面对原片，而是承接 `slot_plan.json` / 候选 shot 这类结构化产物。

更合理的主链路是：

```text
素材理解先行
  -> 文案意图先行
  -> 素材语义强约束
  -> 选中片段后二次重写最终文案
```

也就是：

```text
先理解素材
  -> 得到每段原画面 / 原字幕 / 原口播 / source_meaning
  -> LLM 生成一版分段文案意图
  -> 用每句文案意图去召回、匹配、重打点素材段
  -> 根据实际选中的片段，让 LLM 重写最终配音稿和字幕
  -> TTS / 字幕 / FFmpeg 渲染
```

一句话总结：

```text
不是先写完整成片文案再硬找画面，也不是先把画面拼完再补话术，而是先用文案意图驱动选材，再让最终文案服从已选画面的语义边界。
```

## 2. 为什么不能纯先文案

纯先文案的好处：

- 内容结构完整。
- 开场、卖点、证明、福利、CTA 可以一次性设计。
- TTS 和字幕长度容易控制。
- 更接近直播切片、带货短视频的成片逻辑。

但风险很大：

- 素材不一定有对应画面。
- 画面在展示 A，文案在讲 B。
- 原视频硬字幕和原口播会带着另一套语义，和新文案打架。
- 文案越完整，越容易把素材当“填坑画面”，导致混剪感强、可信度弱。

上一轮 rough v1 的问题就是这个：

```text
先写了一条完整带货脚本
  -> 再把 18 条素材按模板角色塞进时间线
  -> 画面、声音、字幕没有真正同源
```

结果是闭环跑通了，但成片里出现“画面/声音/字幕各玩各的”。

## 3. 为什么不能纯先视频

纯先视频的好处：

- 镜头连续性更自然。
- 不容易切到奇怪动作中间。
- 原素材语义被保留得更多。
- 对 vlog、剧情、活动记录、探店等内容更友好。

但对带货混剪来说也有明显问题：

- 容易变成素材串烧，而不是一条有销售结构的内容。
- 原素材之间的文案语气、卖点顺序、CTA 位置可能互相冲突。
- 每条素材都在讲自己的逻辑，拼起来会稀碎。
- 很难保证最终成片有明确的“钩子 -> 卖点 -> 证明 -> 福利 -> 行动”。

所以原文案不能直接带进最终成片，只能作为“语义锚点”。

## 4. 推荐的混合方案

### 4.1 第一层：素材理解

每个素材先被切成 scene / segment / chunk。

每个片段都要尽量理解出：

```json
{
  "asset_id": "xxx",
  "source_path": "/absolute/path/source.mp4",
  "start": 12.3,
  "end": 16.8,
  "visual_summary": "粉扑按压脸颊，上脸展示妆效",
  "source_ocr": "不卡粉 不浮粉",
  "source_asr": "这个气垫拍开之后很服帖",
  "source_meaning": "强调服帖、不浮粉、上脸自然",
  "product_name": "花西子气垫",
  "selling_points": ["服帖", "不浮粉", "自然妆效"],
  "visual_actions": ["粉扑按压", "上脸试用"],
  "shot_type": "上脸特写",
  "timeline_roles": ["功效证明", "上脸效果"],
  "cta_terms": [],
  "hard_subtitle_risk": "bottom_conflict",
  "audio_reuse": "avoid",
  "voiceover_fit": "good"
}
```

这里最关键的是 `source_meaning`。

它不是原字幕逐字复读，而是把画面、原字幕、原口播统一归纳成这个片段“本来在表达什么”。

### 4.2 第二层：意图文案

LLM 先生成的不是最终配音稿，而是用于找素材的意图文案。

例子：

```json
{
  "script_slot": 3,
  "role": "质地证明",
  "intention_copy": "展示气垫质地轻薄，拍开不厚重",
  "target_duration": 4.0,
  "target_chars": 18,
  "required_visual": ["粉扑", "质地", "手背或近景"],
  "required_meaning": ["轻薄", "不厚重", "服帖"],
  "avoid": ["纯口播无产品", "福利 CTA", "礼盒堆叠"]
}
```

这个阶段的文案可以不丝滑，甚至可以像搜索 query。

它的任务是：

- 规定成片结构。
- 规定每个位置需要什么语义。
- 给向量检索、rerank、重新打点提供目标。

### 4.3 第三层：素材匹配与重打点

用 `intention_copy` 去召回候选片段后，不能直接剪。

还要做 temporal rerank：

```json
{
  "script_slot": 3,
  "candidate_segment_id": "seg_123",
  "match_score": 0.86,
  "usable_start": 13.1,
  "usable_end": 16.4,
  "fit_reason": "这里是粉扑按压手背，画面能证明质地轻薄。",
  "semantic_conflicts": [],
  "cut_risks": ["底部有原硬字幕"],
  "recommended_use": "voiceover_overlay"
}
```

这里的重点不是“这个 chunk 像不像 query”，而是：

- 片段中哪几秒真正可用。
- 是否和该文案位置的语义一致。
- 原字幕/原口播是否会干扰新文案。
- 是否需要避开硬字幕、裁切、遮挡或静音。

### 4.4 第四层：最终文案重写

当时间线片段已经选定后，再让 LLM 统一重写最终配音稿。

输入应该包括：

- 每个 slot 的角色。
- 选中片段的 `visual_summary`。
- 选中片段的 `source_meaning`。
- 原 OCR / ASR。
- 每段时长和目标字数。
- 前后片段语义。

输出：

```json
{
  "slot": 3,
  "final_voice_text": "粉体拍开不是厚重感，是轻薄服帖的奶油质地。",
  "final_subtitle_text": "轻薄服帖，不是厚重假面",
  "target_chars": 18,
  "actual_chars": 22,
  "semantic_alignment": "matches_source_meaning",
  "visual_alignment": "matches_visual"
}
```

这个阶段的原则：

- 保留原片段核心语义。
- 不逐字照搬原文案。
- 不把 A 画面的文案带到 B 画面。
- 字数和片段时长匹配。
- 全片语气统一。
- 卖点顺序和 CTA 收束自然。

## 5. 三层文案字段

后续数据结构里建议固定三层：

### 5.1 `source_meaning`

来源：视频理解。

用途：描述片段原本表达的意思。

例子：

```text
强调气垫上脸服帖、不浮粉，妆效自然。
```

它是素材库的长期资产。

### 5.2 `intention_copy`

来源：LLM 根据目标成片生成。

用途：用于召回和筛选素材。

例子：

```text
展示气垫上脸后服帖自然，不会假面。
```

它是本次任务的临时规划。

### 5.3 `final_voice_text`

来源：LLM 根据已选片段二次重写。

用途：最终 TTS、字幕和成片。

例子：

```text
上脸拍开之后是服帖的奶油肌，不会一眼假面。
```

它是最终成片的内容。

## 6. 和 SentrySearch / SentryMerge 的关系

SentrySearch 的思路是：

```text
视频切成 overlapping chunks
  -> 每个 chunk 做视频 embedding
  -> 用户输入自然语言 query
  -> query embedding 与视频 embedding 匹配
  -> top match 自动 trim 出片段
```

它适合“找事件”，不是完整的广告混剪系统。

值得借鉴的是：

- query / 意图先行。
- chunk + overlap。
- embedding 负责粗召回。
- search 输出 source_file / start_time / end_time / score。
- 自动 trim 只是第一步，不是最终剪辑决策。

SentryMerge 更接近我们要借鉴的后半段：

```text
拿到搜索结果
  -> VLM 判断目标在各机位里的可见起止时间
  -> 精确到子秒重新打点
  -> stitch 成一条片段
```

放到带货混剪里，对应关系是：

```text
intention_copy
  -> 粗召回候选 segment
  -> VLM / Omni temporal rerank
  -> 输出 usable_start / usable_end
  -> 进入最终时间线
```

## 7. 这对后续实现的影响

### 7.1 素材向量化不能只向量化视频画面

至少需要多路向量：

- `video_chunk_embedding`：视频画面本身。
- `clip_summary_embedding`：画面摘要。
- `source_meaning_embedding`：原片段语义。
- `asr_embedding`：原口播。
- `ocr_embedding`：原硬字幕 / 屏幕文字。
- `tag_embedding`：产品、卖点、动作、镜头类型、CTA。

### 7.2 检索不能直接决定剪点

检索只返回候选。

最终必须经过：

```text
候选召回
  -> 语义 rerank
  -> temporal rerank
  -> 可用剪点校验
  -> 字幕/原声风险判断
```

### 7.3 文案生成要分两次

第一次：

```text
生成意图文案，用来规划结构和找素材。
```

第二次：

```text
根据最终选中的素材，重写成片配音稿。
```

这可以避免上一轮的最大问题：文案完整但不贴画面。

## 8. 第一版可执行流程

```text
1. 入库
   - 扫描素材
   - 切 scene / segment / chunk
   - Omni 输出 visual_summary / source_ocr / source_asr / source_meaning
   - 生成多路 embedding

2. 规划
   - 用户输入产品、目标、平台、时长、风格
   - LLM 生成 script_slots
   - 每个 slot 包含 role / intention_copy / target_duration / target_chars

3. 召回
   - 每个 slot 用 intention_copy 检索候选 segment
   - 合并 video / meaning / asr / ocr / tag 多路结果
   - 去重同源素材和重复镜头

4. 重打点
   - Omni / VLM 对候选片段做 temporal rerank
   - 输出 usable_start / usable_end / risks / fit_reason

5. 定稿
   - 固定 timeline slots
   - LLM 根据已选片段重写 final_voice_text / final_subtitle_text
   - 校验字数、时长、语义一致性

6. 渲染
   - TTS
   - 字幕避让或遮挡原硬字幕
   - FFmpeg 切割、拼接、混音、出片
```

## 9. 下一步

下一步先啃“素材向量化”。

但向量化方案必须服务于上面的闭环，不能只做一个视频 embedding demo。

优先解决：

- segment schema。
- `source_meaning` 的生成质量。
- 多路 embedding 字段。
- 检索结果如何合并和 rerank。
- 如何把 `intention_copy` 映射到可剪 `usable_start / usable_end`。
