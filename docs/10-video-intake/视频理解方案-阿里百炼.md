# 视频理解方案：阿里百炼优先验证

## 1. 背景

混剪工作台的基石是“素材理解”。传统路线通常是：

```text
视频
  ↓
切镜头 / 抽关键帧
  ↓
图片理解
  ↓
音频提取
  ↓
Whisper / ASR
  ↓
OCR / 标签 / 摘要
  ↓
片段入库
```

这条路线可控，但成本不低：

- 抽帧数量多时，视觉模型调用成本高。
- Whisper/ASR 需要单独处理音频。
- 画面摘要、字幕、OCR、标签需要拼多段模型调用。
- 片段级结果容易割裂，需要再做聚合。

阿里百炼目前提供了更接近“端到端视频理解”的能力，值得优先验证。

## 2. 阿里方案初步判断

从百炼文档入口看，目前相关能力大致分为两类：

### 2.1 非实时：Qwen-Omni HTTP

定位是通过 OpenAI 兼容接口发起 HTTP 调用，把视频和文本 prompt 一起传给 Qwen-Omni，模型流式返回文本或音频结果。

这类能力适合作为混剪工作台的素材入库主链路。

原因：

- 支持直接理解视频文件中的视觉信息和音频信息。
- 可用结构化 prompt 要求模型输出带时间戳的 storyline、OCR、speaker transcript。
- 素材入库天然不要求实时交互，HTTP 调用更容易缓存、重试和记录原始结果。
- Qwen3.5-Omni 系列支持更长视频，适合先验证素材理解质量。

适合承担：

- 视频内容摘要
- 语义片段级理解
- 人物、场景、事件识别
- 语音转写
- OCR
- 标签提取
- 可检索元数据生成

文档中给出的核心限制：

- 支持地域：北京、新加坡，不同地域 API Key 和 base URL 不同。
- 调用方式：OpenAI 兼容接口，`base_url` 为北京地域 `https://dashscope.aliyuncs.com/compatible-mode/v1`。
- 视频文件输入：`content` 中使用 `{"type": "video_url", "video_url": {"url": "..."}}`。
- 输出模态：支持 `["text"]` 或 `["text", "audio"]`，视频理解入库阶段建议只用 `["text"]`。
- `stream` 必须为 `true`。
- 公网 URL 方式下，Qwen3.5-Omni 系列视频大小限制为 2GB，时长限制为 1 小时。
- Base64 方式下，编码后字符串必须小于 10MB，不适合常规视频素材。
- Qwen3-Omni-Flash 更适合成本敏感的短视频分析，单次音视频限制为 150 秒。

### 2.2 实时：Qwen-Omni / 实时多模态接口

定位更接近“模型实时接收音频与连续图片帧，并交互回答”。

这类能力不适合作为第一版素材入库主链路，但适合做辅助能力。

适合承担：

- 预览时对某个片段追问
- 对候选镜头做二次判断
- 验证某段画面是否符合文案含义
- 在人工剪辑过程中实时询问“这段发生了什么”
- 后续做交互式素材浏览器

不适合第一版作为主链路的原因：

- 实时接口更偏交互，不一定天然产出稳定的批处理结构化结果。
- 需要通过 WebSocket/WebRTC 输入音频和图片帧，不是直接丢一个视频文件。
- 文档建议视频通话场景可以按约 1 张/秒抽帧发送图像，批量素材入库仍然绕回抽帧问题。
- 和素材缓存、去重、重跑机制的耦合更复杂。

## 3. 推荐路线

第一阶段优先验证：

```text
本地视频文件
  ↓
生成可访问的视频 URL，或在小文件场景下转 Base64
  ↓
调用 Qwen-Omni HTTP 视频+文本输入
  ↓
用结构化 Prompt 要求输出 Storyline / Visible Text / Speakers and Transcript
  ↓
转换成统一 Clip Metadata
  ↓
入本地 SQLite/JSON 素材库
```

实时接口暂时作为研究项，不进入 MVP 主链路。

## 5. 需要验证的问题

### 5.1 输入限制

已从文档确认：

- 视频可通过公网 URL 或 Base64 传入。
- Qwen3.5-Omni 系列公网 URL 视频最大 2GB，最长 1 小时。
- Qwen3-Omni-Flash 视频最大 256MB，最长 150 秒。
- Base64 编码后必须小于 10MB。
- 支持 MP4、AVI、MKV、MOV、FLV、WMV 等格式。
- Qwen3.5-Omni 系列公网 URL 最多可传入 512 个视频；Base64 最多 250 个。
- Qwen3-Omni-Flash、Qwen-Omni-Turbo 仅支持输入一个视频文件。

仍需验证：

- 本地素材如何最方便地生成公网 URL。可能需要 OSS、临时文件服务或预签名 URL。
- 模型对竖屏短视频、字幕密集视频、低清素材的理解效果。

### 5.2 输出粒度

文档建议可通过结构化 Prompt 要求：

- 按时间戳输出音视频 storyline。
- 输出可见文字及其时间范围、文字内容、外观特征。
- 输出说话人画像、逐句转写、时间范围、语气、情绪和风格。

仍需验证：

- 是否返回镜头级时间戳？
- 是否返回语义段落级时间戳？
- 是否返回逐句 ASR 时间戳？
- 是否返回 OCR 时间戳？
- 是否返回人物、地点、事件、情绪？
- 是否可以通过 prompt 控制输出结构？

### 5.3 成本和速度

需要确认：

- 计费单位是分钟、token、任务，还是其他？
- 视频视觉信息和音频信息分别如何计费。
- 长视频成本是否可接受？
- 典型 1 分钟、10 分钟、60 分钟视频处理时间分别是多少？
- 是否有并发限制？

### 5.4 可重复性

需要确认：

- 同一视频重复提交结果是否稳定？
- 是否使用固定快照模型，而不是 latest 类模型。
- 返回结果是否包含模型版本、usage、处理参数？
- 是否方便做缓存失效判断？

### 5.5 国内可用性

需要确认：

- 文档、API、SDK 在中国大陆访问是否稳定？
- Python/Node SDK 是否可通过国内镜像安装？
- 是否依赖 OSS、公网 URL 或其他云资源？

## 6. 在混剪工作台里的位置

建议新增一个理解层 Provider 抽象：

```text
VideoUnderstandingProvider
  - analyze(asset) -> raw_result
  - normalize(raw_result) -> Clip[]
```

第一版实现：

```text
AliyunQwenOmniVideoProvider
```

后续可以增加：

```text
FrameAsrFallbackProvider
  - FFmpeg 抽帧
  - 图片模型理解
  - Whisper/ASR
  - OCR
```

这样就不会被单一供应商锁死。

## 8. MVP 验证步骤

建议下一步按这个顺序做：

1. 把阿里百炼视频理解文档中的接口参数整理成 `references/aliyun-video-understanding.md`。
2. 写一个最小脚本，调用 Qwen-Omni HTTP 接口分析一个视频 URL。
3. 保存完整流式文本、usage 和请求参数。
4. 写 normalize 脚本，把模型文本结果转成内部 `Clip[]`。
5. 用 1 个短视频样本验证输出粒度。
6. 再决定是否需要补充 FFmpeg 抽帧 + ASR fallback。

## 9. 当前结论

对于混剪工作台，“视频理解”第一版不建议从抽帧 + Whisper 全链路自建开始。

更合理的路线是：

```text
Qwen-Omni HTTP 非实时视频理解作为主链路
FFmpeg/抽帧/ASR 作为 fallback 和局部补强
实时多模态接口作为后续交互增强
```

这能让我们更快验证核心假设：模型产出的片段理解，是否足够支撑 LLM 做混剪文案和时间线规划。
