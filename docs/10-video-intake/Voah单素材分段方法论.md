# Voah 单素材分段方法论：视觉切点优先，Omni 只做分组

## 结论

对拼贴型带货素材，不能让 Omni 自由决定裁切时间戳。Omni 的语义时间戳会漂移，容易把上一段的口播语义和下一段的画面边界混在一起。更稳的做法是：

1. 先用 ffmpeg scene detection 找视觉切点。
2. 生成候选视觉段 `scene_segments`。
3. 将短于阈值的碎段做方向性合并，当前推荐 `<1.2s` 向后合并。
4. 把候选段 ID、起止时间、时长交给 Omni。
5. 要求 Omni 只能合并相邻候选段，不能自造时间戳。
6. `story_units` 的边界必须来自候选视觉段边界。
7. 后续召回与文案以 `story_units` 为主；最终剪辑仍可在其子段中截取。

## 为什么这样做

- 原片如果本身很碎，强行合成少量长段会把不同人物、不同场景、不同产品对象混到一起。
- Omni 自由分段适合生成标签、语义描述和角色判断，不适合直接提供裁切边界。
- 视觉切点负责“边界干净”，Omni 负责“哪些相邻视觉段属于同一段语义”。

## 当前推荐参数

- scene detection threshold：`0.36` 起步。
- raw visual segment 最小时长：`0.25s`。
- 可用候选段合并阈值：`1.2s`。
- 小碎段合并方向：优先向后合并，避免污染前一段已经干净结束的边界。
- Omni 参数：`temperature=0.1`，`top_p=0.8`。

## 关键约束

Omni prompt 必须包含：

- 只能按候选段 ID 分组。
- 只允许合并相邻段。
- start/end 必须等于首尾候选段的 start/end。
- 画面换人、换场、换对象、空镜到人物、真人到产品特写，一般不要合并。
- 输出 `scene_segment_ids`。

## 质量检查

每次新分段至少检查：

- contact sheet：确认每段中帧代表不同视觉内容。
- 相邻边界帧：检查前段末帧和后段首帧是否粘连。
- 过短段数量：`<1.2s` 的段是否过多。
- 边界来源：所有 `story_units` 边界是否来自视觉候选段。

## 当前发现的新问题

人工检查发现：每个 shot 结束处疑似粘了一帧下一个 shot 的首帧。这个现象非常规律，可能是 ffmpeg 裁切时 end 边界包含、时间戳取整、B-frame/keyframe 或 re-encode 帧选择导致。需要用末帧/首帧抽取做帧级验证，并在裁切时考虑 `end_epsilon` 或帧精度裁切。

## 帧级边界检查补充

人工复核 `001_001.833-004.867.mp4` 后确认：使用 `ffmpeg -ss start -t duration` 重编码时，前一段末尾确实可能多吃到下一段首帧。该 clip 标称 `1.833-4.867s`，理论约 91 帧，但实际写出了 92 帧，最后一帧是下一段产品特写。

复现实验：

- 原裁切：92 帧，末帧粘到下一段。
- `duration - 0.5/fps`：91 帧，末帧干净。
- `duration - 1/fps`：91 帧，末帧干净。
- `duration - 2/fps`：90 帧，过度扣帧风险更高。

工程结论：

- 当前裁切命令不应直接使用 `-t end-start`。
- 对视觉切点裁切，默认使用半开区间 `[start, end)`，实现上应在 end 侧减去 `end_epsilon`。
- 推荐 `end_epsilon = 0.5 / fps` 起步；如果仍粘帧，再提高到 `1 / fps`。
- 更稳的长期方案是帧号裁切：`start_frame = ceil(start*fps)`，`end_frame = floor(end*fps)`，用 `trim=start_frame=...:end_frame=...`。
- QA 必须抽每段末帧，检查是否等于下一段首帧/切点帧。

保留另一个发现：如果相邻边界本来视觉差异极低，即使不粘帧，人眼也会感觉接缝不明显。这类边界仍可标记为 `low_visual_delta_boundary`，作为合并候选。
