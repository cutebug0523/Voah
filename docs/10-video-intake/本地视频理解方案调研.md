# 本地视频理解方案调研

调研日期：2026-06-01

## 1. 先界定“本地”

这里把“本地”拆成两种口径：

1. **本地文件优先**：原始素材在本机，允许把压缩代理或 Base64 发给云模型分析。
2. **完全本地/离线优先**：视频、音频、字幕、抽帧、识别、向量化都在本机完成，不依赖云 API。

如果只是“本地文件优先”，上一轮已经验证过：

```text
本地原片 -> 压缩代理 -> Base64 -> 阿里百炼 qwen3.5-omni-plus -> 结构化理解
```

这条最快，但严格来说不是完全本地。

如果用户明确说“就得本地”，第一版应按完全本地路线设计。

## 2. 候选方案

### 2.1 方案 A：全本地分层流水线

流程：

```text
本地视频
  -> ffprobe 读取元信息
  -> PySceneDetect / FFmpeg 做镜头或语义候选切分
  -> FFmpeg 抽关键帧 / 低帧率代理
  -> whisper.cpp 或 faster-whisper 做 ASR
  -> PaddleOCR 做硬字幕 OCR 和字幕区域检测
  -> Qwen3-VL / Qwen2.5-VL / MiniCPM-V 对关键帧或短片段做视觉理解
  -> 本地 LLM 聚合成 Clip Metadata
  -> SQLite/JSONL + FAISS 入库
```

优点：

- 真正本地，素材不离机。
- 每一层都能缓存，后续可以逐步替换模型。
- ASR、OCR、画面理解互不绑死，方便做质量对比。
- 适合 Codex skill/MCP 化：每个步骤都能封装成一个小工具。
- 对当前口红、气垫这类竖屏带货素材很合适，因为硬字幕和口播本身就是强信号。

缺点：

- 组件多，第一版要做编排和缓存。
- 视觉模型对“连续动作”的理解弱于端到端视频模型，需要按片段采样和聚合。
- 模型输出时间戳需要我们自己约束和校正。

我建议第一版选这个。

### 2.2 方案 B：本地端到端 Omni 模型

代表：

- Qwen2.5-Omni 3B/7B 本地部署。
- 后续如果有更轻的开源 Omni 模型，也可接入。

流程：

```text
本地视频 -> Qwen2.5-Omni 本地推理 -> 结构化理解
```

优点：

- 形态最接近阿里百炼 Qwen-Omni。
- 理论上可以同时利用音频、视频、文字。
- 后续可做实时交互式素材问答。

缺点：

- 本地部署重，通常更适合 NVIDIA GPU / vLLM / Docker 路线。
- 在 Mac 上能跑不等于适合批量素材入库。
- 输出稳定性、ASR 时间戳、OCR 区域等仍需工程补强。
- 第一版会把大量精力消耗在模型部署上，而不是素材库闭环。

结论：暂不作为 MVP 主链路，可作为后续替换后端。

### 2.3 方案 C：本地 VLM 直接读视频 + 单独 ASR/OCR

代表：

- Qwen3-VL：官方说明支持本地视频路径、fps、像素预算和视频 token 控制。
- Qwen2.5-VL：支持视频理解，Transformers 文档提到动态 FPS 采样与视频理解能力。
- MiniCPM-V：轻量多模态路线，适合低资源设备试验。

流程：

```text
本地视频 -> Qwen3-VL / Qwen2.5-VL 视觉理解
本地音频 -> whisper.cpp / faster-whisper
关键帧 -> PaddleOCR
三路结果 -> 聚合 JSON
```

优点：

- 比方案 A 更少手写视觉聚合逻辑。
- 仍然保持本地。
- Qwen3-VL 的本地视频输入已经比较贴近我们要的“视频理解”。

缺点：

- 仍不解决音频转写，ASR 需要单独做。
- 直接喂整段视频会吃内存，需要控制 fps、分辨率和总像素预算。
- 对 Mac 本地推理速度要实际压测。

结论：可以作为方案 A 里的视觉理解后端，而不是单独做一套架构。

### 2.4 方案 D：传统视觉 + ASR/OCR，不上 VLM

流程：

```text
镜头切分 -> ASR -> OCR -> CLIP/SigLIP 图像向量 -> 文本向量 -> 检索库
```

优点：

- 最轻、最快、最稳。
- 适合先做素材去重、字幕提取、口播检索。

缺点：

- “画面里发生了什么”理解不足。
- 不能稳定生成混剪所需的镜头语义，如情绪、动作、产品露出方式、可用段落。

结论：可做降级模式，不够当基石。

### 2.5 方案 E：本地公网/内网代理 + 云模型

流程：

```text
本地视频 -> 压缩代理 -> 临时 URL / Base64 -> 云端 Omni -> 结果回写本地
```

优点：

- 质量高，已经用 qwen3.5-omni-plus 跑通过。
- 不需要本地 GPU。

缺点：

- 不满足完全本地。
- Base64 有 10MB 限制，长视频要走 URL/OSS/临时文件服务。
- 批量处理有成本。

结论：可以作为质量对照组，不作为“就得本地”的主方案。

## 3. 我挑的路线

第一版选：

```text
方案 A：全本地分层流水线
```

但具体落地时拆成两阶段：

### 3.1 第一步：不碰大模型部署，先建本地理解骨架

先实现这些确定性能力：

```text
scan_assets
probe_video
fingerprint_asset
detect_scenes
extract_keyframes
extract_audio
```

产物：

```text
cache/assets/{asset_id}/metadata.json
cache/assets/{asset_id}/scenes.json
cache/assets/{asset_id}/keyframes/*.jpg
```

这一步跑完后，即使没有 VLM，也已经有素材清单、时长、分辨率、镜头边界、关键帧。

### 3.2 第二步：接本地 ASR/OCR/VLM

推荐优先级：

```text
ASR: whisper.cpp
OCR: PaddleOCR PP-OCRv5
视觉理解: Qwen3-VL 小模型优先，跑不动再退到 Qwen2.5-VL / MiniCPM-V / 抽帧图片理解
向量库: SQLite + FAISS
```

原因：

- 当前是 darwin/macOS，whisper.cpp 对 Apple Silicon/Metal/Core ML 友好。
- PaddleOCR 的中文硬字幕识别成熟，适合你的素材。
- Qwen3-VL 官方支持视频本地路径、采样 fps 和像素预算；视觉理解可以先低帧率跑，不需要一上来理解原片全帧。
- SQLite + FAISS 足够本地个人素材库，不急着上独立向量数据库服务。

## 5. 和 Codex Skill / MCP 的拆法

先不做大 MCP，一开始保留脚本，等命令稳定后再 MCP 化。

推荐工具边界：

```text
scripts/media_probe.py
  - ffprobe 元信息
  - hash

scripts/media_scenes.py
  - PySceneDetect / FFmpeg 镜头切分
  - 关键帧抽取

scripts/media_asr.py
  - whisper.cpp / faster-whisper

scripts/media_ocr.py
  - PaddleOCR

scripts/media_understand_local.py
  - Qwen3-VL / Qwen2.5-VL / MiniCPM-V adapter

scripts/media_normalize.py
  - 合并 scene/asr/ocr/vision
  - 输出 clip metadata
```

稳定后再包成 MCP：

```text
media.scan_assets
media.analyze_asset_local
media.search_clips
media.get_clip
```

## 6. 对当前素材的判断

`/Users/noah/混剪/口红/口红` 里都是 1080x1920 的竖屏短视频，大多 18-73 秒、24-92MB。

这类素材的理解优先级应是：

1. **ASR/硬字幕 OCR**：带货口播信息密度最高，直接决定可检索性。
2. **镜头切分/关键帧**：产品展示、唇部特写、试色、礼盒开箱要拆出来。
3. **视觉标签**：人物口播、产品露出、上嘴演示、手背试色、礼盒 CTA。
4. **字幕区域**：后续重剪时决定 keep/crop/cover/avoid。

因此不建议第一步就追求“端到端本地 Omni”。先把可缓存的本地流水线跑通，更快变成可复用资产库。
