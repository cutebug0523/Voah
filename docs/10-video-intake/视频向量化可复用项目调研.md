# 视频向量化可复用项目调研

调研日期：2026-06-02

## 1. 结论

有可复用项目，但没有一个能原封不动解决我们的“混剪素材库”。

最接近的是：

```text
SentrySearch
```

它已经实现了：

```text
视频目录
  -> 分块 chunk
  -> 用视频 embedding 模型把每个 chunk 向量化
  -> 存 ChromaDB
  -> 文本或图片查询
  -> 返回匹配时间段
  -> ffmpeg 自动裁出片段
```

这和我们讨论的“视频向量化 + rerank 打时间 + 代码分割”非常接近。

但它偏“监控/行车记录仪事件搜索”，不偏“短视频混剪素材库”。所以建议：

```text
不整包迁入
重点复用/参考它的分块、索引、搜索、裁片实现
我们自己保留 clip metadata、Omni 理解、字幕策略、timeline 生成
```

## 2. 可复用项目清单

### 2.1 SentrySearch

GitHub：

```text
https://github.com/ssrajadh/sentrysearch
```

定位：

```text
Semantic search over video footage. Type what you're looking for, get a trimmed clip back.
```

核心流程：

- 把视频切成重叠 chunks。
- 每个 chunk 直接作为视频送进 embedding 模型。
- 支持 Gemini Embedding、Alibaba DashScope `qwen-cloud`、本地 Qwen3-VL。
- 向量存在本地 ChromaDB。
- 查询时把文本或图片嵌入到同一向量空间。
- 返回 top matches。
- 用 ffmpeg 从原视频自动裁出匹配片段。

特别值得注意：

- 已经支持 `qwen-cloud`，默认模型是 `qwen3-vl-embedding`。
- README 说明本地 chunk 会通过 DashScope 官方 SDK 上传到托管临时 OSS，再给 API 消费。
- 有 `--chunk-duration`、`--overlap`、`--target-resolution`、`--target-fps` 这些我们需要的参数。
- 搜索支持 `--threshold`、`--save-top`、`--dedupe`、`--no-trim`。
- 支持 image search。
- Apache-2.0 license。

可复用程度：

```text
高
```

适合我们借：

- chunk 数据结构。
- 本地索引结构。
- ChromaDB 存储方式。
- DashScope embedding 上传/调用方式。
- 搜索结果去重。
- ffmpeg 自动裁片。

不适合直接照搬的地方：

- 它的 chunk 粗粒度默认偏 30s，混剪通常需要 1-6s 的镜头级/语义级片段。
- 它主要按视觉事件搜，不包含 Omni 输出的 ASR/OCR/混剪用途标签。
- 它的 rerank 还是检索排序，不等于我们要的 temporal rerank + 剪辑可用性判断。
- 它不是素材创作系统，不负责文案、叙事结构、配音字幕和 timeline。

建议：

```text
先把 SentrySearch 当参考实现。
后续可以单独试跑 qwen-cloud backend，看它的视频 embedding 对当前口红素材检索效果如何。
```

### 2.2 Qwen3-VL-Embedding / Qwen3-VL-Reranker

GitHub：

```text
https://github.com/QwenLM/Qwen3-VL-Embedding
```

官方百炼文档：

```text
https://help.aliyun.com/zh/model-studio/multimodal-embedding-api-reference
https://help.aliyun.com/zh/model-studio/rerank
```

定位：

- `qwen3-vl-embedding`：多模态向量模型。
- `qwen3-vl-rerank`：多模态重排序模型。

官方能力要点：

- 支持 text / image / video。
- 支持独立向量和融合向量。
- `qwen3-vl-embedding` 支持 2560、2048、1536、1024、768、512、256 维。
- 百炼视频 embedding 限制：视频文件不超过 50MB。
- 视频支持 MP4、AVI、MOV，视频仅支持 URL。
- `qwen3-vl-rerank` 支持文本、图片、视频混合排序。
- `qwen3-vl-rerank` 单次文档限制大致是文本 100、图片 40、视频 4。

可复用程度：

```text
高
```

这不是一个现成业务系统，但它是我们该选的模型层。

建议第一版模型组合：

```text
召回 embedding: qwen3-vl-embedding
排序 rerank: qwen3-vl-rerank
理解/归一化: qwen3.5-omni-plus
```

其中：

- `qwen3.5-omni-plus` 负责详细理解、ASR/OCR、混剪语义。
- `qwen3-vl-embedding` 负责把片段变成可召回向量。
- `qwen3-vl-rerank` 负责对候选片段做二次排序。
- 最终 temporal rerank 仍建议用 Omni/LLM 做“重新打点 + 剪辑判断”。

### 2.3 Lighthouse

GitHub：

```text
https://github.com/line/lighthouse
```

定位：

```text
video moment retrieval and highlight detection
```

它不是普通向量库，而是专门做“给一段文本 query，在视频里找相关时间窗口”的 moment retrieval。

可复用程度：

```text
中
```

优点：

- 输出就是 `pred_relevant_windows: [[start, end, score], ...]`，很接近我们的 temporal rerank。
- 支持 video moment retrieval、highlight detection、audio moment retrieval。
- 有 inference API 和 Gradio demo。

缺点：

- 偏学术/benchmark。
- 依赖预训练权重和特征提取，如 CLIP/SlowFast/PANNs。
- README 提到当前最大视频时长按 benchmark 约束为 150s。
- 和我们云端 Omni/Qwen embedding 路线不是天然一体。

建议：

```text
先不作为主链路。
后续研究 temporal rerank 时，可以参考它的输出结构和评估方式。
```

可复用程度：

```text
中
```

优点：

- 产品方向和“视频编辑素材库”很接近。
- 结构上有 Web、后台任务、ML 服务、PostgreSQL、ChromaDB。
- 功能包含 transcription、object/text detection、scene analysis。

缺点：

- README 自称还在 active development，不是 production-ready。

- License 不是简洁的 Apache/MIT 路线，需要谨慎。
- 它更像完整应用，不适合作为 Codex skill/MCP 的轻量内核。

建议：

```text

```

## 3. 向量数据库选择

### 3.1 ChromaDB

GitHub：

```text
https://github.com/chroma-core/chroma
```

优点：

- SentrySearch、Edit Mind 都在用。
- 本地持久化简单。
- Python 调用容易。
- 足够第一版。

缺点：

- 后续如果要非常大的素材库、复杂过滤、服务化部署，可能需要再评估。

结论：

```text
第一版可以用 ChromaDB。
```

### 3.2 LanceDB

GitHub：

```text
https://github.com/lancedb/lancedb
```

优点：

- 面向多模态 AI 的嵌入式检索库。
- 对本地文件/表格式数据友好。

结论：

```text
也适合第一版，但 SentrySearch 生态更偏 ChromaDB，先沿用 ChromaDB 阻力最小。
```

### 3.3 Qdrant / FAISS

GitHub：

```text
https://github.com/qdrant/qdrant
https://github.com/facebookresearch/faiss
```

判断：

- Qdrant 更适合服务化、过滤复杂、规模更大的情况。
- FAISS 更像底层向量检索库，需要自己补 metadata/持久化/过滤。

结论：

```text
MVP 不优先。
```

## 4. 推荐路线

第一版不要自己从零搓完整视频向量化。

建议按这个顺序：

```text
1. 试跑 SentrySearch 的 qwen-cloud backend
2. 用当前口红素材做 5-10 条 query 测召回
3. 记录它的 chunk、embedding、ChromaDB、裁片产物
4. 决定是 fork/adapt，还是只抄结构自己写轻量版
5. 我们自己的系统里保留 Omni 理解结果作为强 metadata
```

候选融合架构：

```text
原视频
  -> Omni 理解：summary / segments / asr / ocr / tags / subtitle_region
  -> Qwen3-VL-Embedding：asset / chunk / clip / keyframe 向量
  -> ChromaDB：向量 + metadata
  -> query 召回 Top-K
  -> qwen3-vl-rerank 或 Omni：rerank
  -> Omni/LLM：temporal rerank，重新打 usable_start / usable_end
  -> FFmpeg：切割
```

## 5. 我们不该直接外包给现成项目的原因

现成项目解决的是：

```text
给我一句话，帮我在视频库里找某个时刻。
```

我们要解决的是：

```text
给我一个创作目标，帮我从素材库里挑片段、组织叙事、避开硬字幕、配音上字幕、生成可执行时间线并渲染成片。
```

所以视频向量化只是底层召回能力。它可以复用，但不能决定整个架构。

### 7.3 实测后的工程结论

SentrySearch 值得复用的部分：

```text
1. DashScope SDK 上传本地 chunk 到临时 OSS 的方式
2. chunk + overlap 参数化
3. preprocess 到低分辨率/低 fps 后再 embedding
4. ChromaDB collection 设计
5. search 输出 source_file/start_time/end_time/score
6. ffmpeg trim_top_results 裁片逻辑
7. TLS/网络失败重试
```

不建议直接照搬的部分：

```text
1. 只做纯视频 chunk embedding
2. 默认 30 秒 chunk
3. metadata 只有 source_file/start/end
4. 搜索结果直接裁片，不经过混剪语义 rerank
```

我们自己的第一版应改成：

```text
Omni 理解结果
  -> 生成 scene/segment 级 metadata
  -> 对 video_chunk、summary、asr、ocr、tags 分别向量化
  -> ChromaDB/SQLite 保存向量和 metadata
  -> query 粗召回 topK
  -> qwen3-vl-rerank 或 Omni temporal rerank
  -> 输出 usable_start / usable_end
  -> FFmpeg 裁片
```
