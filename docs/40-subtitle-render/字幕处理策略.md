# 字幕处理策略

## 1. 问题定义

当前素材大概率已经自带字幕，而且多数是“硬字幕”。

硬字幕不是独立字幕轨，而是已经烙进视频画面的像素。它和人物、背景、物体一样，属于画面内容的一部分。

因此所谓“替换字幕”分两种情况：

- **软字幕**：视频里有独立字幕轨，或有 `.srt` / `.ass` 文件。可以关闭、替换或重新渲染。
- **硬字幕**：字幕已经烧录到画面里。不能直接删除，只能遮挡、裁剪、重排或用视频修复模型尝试去除。

第一版系统应默认按硬字幕处理。

## 2. 结论

第一版不要把目标定成“自动替换原字幕”。

更合理的目标是：

```text
识别原字幕位置
  ↓
判断它是否影响新字幕
  ↓
选择处理策略
  ↓
生成新字幕安全区
```

也就是：先把“原字幕”当成画面约束，而不是当成可编辑字幕。

## 3. 可选策略

### 3.1 保留原字幕

适用场景：

- 原字幕和画面内容相关。
- 原字幕不影响新配音字幕。
- 混剪保留原对白或原剧情信息。

优点：

- 成本最低。
- 无画质损伤。
- 最稳定。

缺点：

- 新配音字幕和原字幕可能信息冲突。
- 画面容易显得杂乱。

### 3.2 避让原字幕

适用场景：

- 原字幕在画面底部。
- 新字幕可以放在中部、上方或侧边。
- 成片是短视频平台样式，可以接受字幕不在最底部。

做法：

- OCR 检测原字幕区域。
- 给新字幕设置安全区，避免覆盖同一区域。
- 对竖屏成片，可以把新字幕放在画面中下方但高于原字幕。

优点：

- 不破坏画面。
- 实现简单。

缺点：

- 屏幕上会同时存在两套字幕。

### 3.3 裁剪掉原字幕

适用场景：

- 原字幕在底部边缘。
- 裁剪后不影响主体人物和关键画面。
- 横屏素材转竖屏时本来就需要重构画幅。

做法：

- 检测硬字幕区域。
- 在 `crop` / `scale` 时避开底部字幕。
- 或者把视频缩放进画布，上下留黑/模糊背景区，再在安全区放新字幕。

优点：

- 成片更干净。
- 比修复字幕稳定。

缺点：

- 可能损失画面信息。
- 字幕位置太靠中时不可用。

### 3.4 遮挡原字幕

适用场景：

- 原字幕区域固定且底部背景不重要。
- 成片风格允许底部有字幕条、渐变条或遮罩。

做法：

- 在原字幕区域加半透明黑条、模糊条或底部信息栏。
- 新字幕放在遮挡区域上方或遮挡条内。

优点：

- 稳定、便宜、可控。
- 短视频里接受度较高。

缺点：

- 会牺牲画面质感。
- 如果原字幕很高或位置变化大，会遮住人物。

### 3.5 模糊原字幕

适用场景：

- 原字幕必须弱化，但不需要完全消失。
- 画面底部不是重点。

做法：

- 对检测出的字幕区域做局部 blur。
- 新字幕盖在 blur 区域上方。

优点：

- 比纯黑条自然一点。
- FFmpeg 可实现。

缺点：

- 仍能看出处理痕迹。
- 动态字幕、花字、描边字幕效果一般。

### 3.6 视频修复 / Inpainting 去字幕

适用场景：

- 对画质要求高。
- 原字幕严重干扰画面。
- 可以接受较高计算成本和失败率。

做法：

- OCR/检测字幕 mask。
- 用视频 inpainting 模型逐帧修复。
- 再做时序一致性处理，降低闪烁。

优点：

- 理想情况下最干净。

缺点：

- 成本高。
- 慢。
- 容易闪烁。
- 对复杂背景、人物遮挡、动态镜头不稳定。

第一版不建议做成默认能力。

## 4. 推荐 MVP 策略

第一版建议只支持三种稳定策略：

```text
keep      保留原字幕，新字幕避让
crop      裁剪掉原字幕
cover     遮挡原字幕并重上新字幕
```

默认策略：

```text
auto
```

`auto` 的决策逻辑：

1. 如果检测不到明显硬字幕，正常生成新字幕。
2. 如果硬字幕位于底部 20% 以内，优先尝试裁剪或避让。
3. 如果裁剪会切掉人物脸部或关键主体，改用遮挡。
4. 如果原字幕位置不稳定或覆盖主体，保留原字幕并把新字幕移到上方。
5. 不自动使用 inpainting。

## 5. 视频理解阶段要新增的字段

素材入库时需要记录硬字幕信息。

```json
{
  "burned_subtitle": {
    "detected": true,
    "language": "zh",
    "regions": [
      {
        "start": 0.0,
        "end": 12.5,
        "box": {
          "x": 84,
          "y": 890,
          "width": 910,
          "height": 120
        },
        "text": "你到底有没有想过我的感受",
        "confidence": 0.86
      }
    ],
    "position": "bottom",
    "style": {
      "color": "white",
      "outline": true,
      "background": false
    }
  }
}
```

这些信息来自：

- Qwen-Omni 的可见文字输出。
- OCR。
- 抽关键帧后的文本区域检测。

第一版可以先只记录：

```json
{
  "detected": true,
  "position": "bottom",
  "safe_area": "middle"
}
```

## 6. 时间线要新增的字段

Timeline 里需要记录字幕策略。

```json
{
  "subtitle_policy": {
    "source_hard_subtitle": "auto",
    "generated_subtitle": true,
    "placement": "avoid_source_subtitle",
    "fallback": "cover"
  }
}
```

单个视频片段也可以覆盖策略：

```json
{
  "clip_id": "asset_001_c0007",
  "source_in": 35.2,
  "source_out": 38.7,
  "target_in": 0.0,
  "target_out": 3.5,
  "subtitle_policy": {
    "source_hard_subtitle": "crop",
    "generated_subtitle_position": "middle_lower"
  }
}
```

## 7. FFmpeg 可实现能力

第一版 FFmpeg 可以支持：

- 裁剪：`crop`
- 缩放：`scale`
- 加黑条：`drawbox`
- 局部模糊：`boxblur` + mask/crop/overlay
- 烧录新字幕：`subtitles` 或 `ass`
- 背景虚化画布：常见横屏转竖屏方案

FFmpeg 不负责：

- 准确识别字幕位置。
- 理解原字幕内容。
- 智能修复被字幕挡住的背景。

## 8. 和混剪风格的关系

如果素材本身带字幕，混剪策略要更像短视频二创：

- 原字幕可以保留为“画面证据”。
- 新配音字幕不一定要在最底部。
- 可用统一字幕条建立成片风格。
- 对关键信息片段，可以让原字幕留在画面中，新字幕只承担旁白节奏。
- 对强剧情素材，原字幕 OCR 反而可以作为检索依据。

## 9. 实现顺序

建议顺序：

1. 在视频理解 Prompt 中要求模型输出可见文字和时间戳。
2. 在 Clip Metadata 中加入 `burned_subtitle` 字段。
3. 做一个人工可选的字幕策略：`keep` / `crop` / `cover`。
4. 在渲染脚本中支持 `crop` 和 `cover`。
5. 后续再做自动字幕区域检测。
6. 最后再评估是否需要 inpainting 去字幕。

## 10. 当前决策

- 第一版不承诺自动去除硬字幕。
- 第一版优先把硬字幕作为素材理解和画面布局约束。
- 第一版新字幕默认避让或遮挡，不做像素级修复。
- 原字幕 OCR 是素材检索的重要信号，应保留入库。
